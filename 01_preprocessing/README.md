# EEG-TBI Preprocessing Pipeline

Пайплайн предобработки и контроля качества ЭЭГ-данных для исследования последствий черепно-мозговой травмы (ЧМТ).

Репозиторий содержит код, описание методики и безопасные агрегированные примеры результатов. Сырые ЭЭГ-сигналы, полные файлы эпох, клинические метаданные и приватные пути к данным не публикуются.

## Место в проекте

Этот раздел соответствует первому этапу исследования:

1. [`01_preprocessing/`](../01_preprocessing/) — предобработка ЭЭГ и контроль качества.
2. [`02_quantitative_eeg_features/`](../02_quantitative_eeg_features/) — расчёт количественных признаков ЭЭГ.
3. [`03_machine_learning/`](../03_machine_learning/) — модели классификации и оценка информативности признаков.

## О проекте

Цель данного этапа — привести разнородные ЭЭГ-записи пациентов с ЧМТ и контрольной группы к единому техническому формату для последующего количественного анализа и машинного обучения.

Пайплайн решает задачу **подготовки данных**, а не полного анализа ЭЭГ. Он формирует стандартизированные записи, выполняет автоматизированную очистку артефактов, применяет контроль качества, унифицирует каналы и сохраняет таблицу записей, пригодных для дальнейшей обработки.

Основной результат этапа:

```text
analysis_ready_inventory_cleaned.csv
```

Этот файл используется как вход для следующего этапа — расчёта спектральных, временных, событийных и сетевых признаков ЭЭГ.

---

## Назначение пайплайна

Пайплайн выполняет следующие этапы:

1. формирование inventory исходных записей;
2. загрузку клинических `.mat`-файлов ЧМТ и контрольных `.edf`-файлов;
3. стандартизацию частоты дискретизации, фильтрации, референса и длительности эпох;
4. автоматическое выявление плохих каналов;
5. интерполяцию плохих каналов;
6. очистку артефактов с использованием ICA/ICLabel;
7. коррекцию и отбраковку эпох с помощью AutoReject;
8. финальный record-level cleaning-QC;
9. дополнительный амплитудный QC-аудит;
10. унификацию каналов между группами;
11. сохранение итоговой таблицы `analysis_ready_inventory_cleaned.csv`.

Общая схема:

```text
локальные исходные файлы
        ↓
inventory
        ↓
базовая стандартизация MNE Epochs
        ↓
bad channels detection + interpolation
        ↓
ICA / ICLabel
        ↓
AutoReject
        ↓
финальный record-level cleaning-QC
        ↓
амплитудный QC-аудит
        ↓
унификация каналов
        ↓
analysis_ready_inventory_cleaned.csv
```

## Данные

Пайплайн использует два источника данных.

### Группа ЧМТ

Клинические ЭЭГ-записи пациентов с черепно-мозговой травмой представлены локальными файлами MATLAB/EEGLAB:

```text
/path/to/local/tbi_mat_files/
    *.mat
```

Ожидается, что внутри `.mat`-файлов содержится структура `EEG`, включающая, при наличии:

- `EEG.data` — массив ЭЭГ-сигнала;
- `EEG.srate` — частоту дискретизации;
- `EEG.chanlocs` — информацию о каналах;
- `EEG.bad_chans` — предварительно размеченные плохие каналы;
- `EEG.bad_epochs` — предварительно размеченные плохие эпохи.

Клинические данные являются закрытыми исследовательскими данными. Они были обезличены и использовались только в рамках исследования, но не могут быть опубликованы из-за институциональных ограничений, требований конфиденциальности и условий доступа.

### Контрольная группа

Контрольные записи берутся из открытого датасета OpenNeuro `ds005385` в формате EDF:

```text
/path/to/local/openneuro/ds005385/
    sub-*/ses-*/eeg/*.edf
```

В репозитории EDF-файлы не дублируются, так как они занимают много места и доступны из внешнего источника. Для запуска пайплайна пользователь должен самостоятельно скачать датасет и указать локальный путь к нему.

## Запуск

1. Клонировать репозиторий:

```bash
git clone https://github.com/L-Gaysina/eeg-tbi.git
cd eeg-tbi/eeg-tbi-preprocessing
```

2. Установить зависимости:

```bash
pip install -r requirements.txt
```

3. Скачать контрольный датасет OpenNeuro `ds005385` и сохранить его локально.

4. Подготовить локальную папку с клиническими `.mat`-файлами ЧМТ.

5. Открыть основной ноутбук:

```text
notebooks/01_prepare_eeg_dataset.ipynb
```

6. Указать локальные пути к данным в `CONFIG`.

7. Запустить ноутбук сверху вниз.

## Запуск в Google Colab

Ноутбук можно запускать в Google Colab. В этом случае необходимо:

1. подключить Google Drive;
2. указать пути к папкам с `.mat` и EDF-файлами;
3. убедиться, что сырые данные не находятся внутри папки репозитория;
4. сохранять рабочие результаты в отдельную папку, не предназначенную для коммита в GitHub;
5. перед публикацией очистить outputs ячеек от локальных путей и приватной информации.

Массовая обработка реализована в checkpoint/resume-режиме:

```text
после каждой успешно обработанной записи сохраняется inventory
при повторном запуске уже обработанные записи пропускаются
ошибки обработки сохраняются отдельно
```

Это важно для длительной обработки EEG-файлов в Google Colab, где сессия может быть прервана.

## Очистка артефактов

После базовой стандартизации выполняется автоматизированная очистка данных.

### Bad channel detection

Для каждого канала оценивается дисперсия сигнала. Канал помечается как потенциально плохой, если его дисперсия превышает медианную дисперсию по каналам более чем в заданное число раз.

Плохие каналы интерполируются с использованием топологической информации о расположении электродов.

### ICA / ICLabel

Для удаления глазодвигательных, мышечных, кардиальных и технических артефактов выполняется ICA-разложение.

ICA обучается на копии данных с high-pass фильтрацией 1 Гц. После обучения компоненты классифицируются с помощью ICLabel.

Компоненты исключаются, если они относятся к одному из артефактных классов и вероятность классификации превышает заданный порог.

Артефактные классы:

```text
eye blink
heart beat
muscle artifact
line noise
channel noise
other
```

### AutoReject

После ICA применяется AutoReject. Этот этап позволяет:

- локально интерполировать повреждённые каналы на уровне отдельных эпох;
- исключать чрезмерно артефактные эпохи;
- сохранить больше данных по сравнению с жёстким глобальным амплитудным порогом.

Для каждой записи сохраняются QC-показатели:

- число найденных плохих каналов;
- список интерполированных каналов;
- число удалённых ICA-компонент;
- список удалённых компонент;
- число эпох до AutoReject;
- число эпох после AutoReject;
- доля эпох, отклонённых AutoReject.

## Финальный контроль качества

После ICA/ICLabel и AutoReject выполняется финальный record-level cleaning-QC.

Запись исключается из дальнейшего анализа, если выполняется хотя бы одно из условий:

1. число эпох после очистки меньше 20;
2. доля эпох, отклонённых AutoReject, больше 0.50;
3. число плохих каналов больше 16.

Итог после финального QC:

| Группа | Записей до QC | Записей после QC | Исключено | Субъектов после QC | Эпох после QC |
|---|---:|---:|---:|---:|---:|
| Control | 364 | 363 | 1 | 72 | 16 387 |
| TBI | 215 | 212 | 3 | 92 | 23 832 |
| **Итого** | **579** | **575** | **4** | **164** | **40 219** |

## Унификация каналов

До унификации группы имели разное число каналов:

| Этап | Control | TBI | Общий набор |
|---|---:|---:|---:|
| Число каналов после стандартизации | 64 | 60 | — |
| Число каналов после пересечения групп | 58 | 58 | 58 |

Все дальнейшие этапы анализа используют только единый набор каналов. Это необходимо для корректного расчёта признаков и сопоставимости групп.

Список общих каналов сохраняется в файле:

```text
outputs_example/tables/common_channels.txt
```

## Амплитудный QC-аудит

После финального cleaning-QC дополнительно выполняется амплитудный контроль качества.

Для каждой записи рассчитываются:

- медианная peak-to-peak амплитуда;
- 95-й перцентиль peak-to-peak амплитуды;
- максимальная peak-to-peak амплитуда;
- доля эпох, превышающих p2p-порог.

Порог p2p-амплитуды:

```text
300 мкВ
```

Амплитудный QC используется как дополнительный аудит качества после ICA/ICLabel и AutoReject, а не как основной автоматический фильтр.

Сводка амплитудного QC:

| Группа | Записей | Медианная p2p-амплитуда, мкВ | Медианный 95-й перцентиль p2p, мкВ | Медианная доля эпох выше p2p-порога | Максимальная доля эпох выше p2p-порога |
|---|---:|---:|---:|---:|---:|
| Control | 363 | 62.55 | 94.00 | 0.000 | 0.138 |
| TBI | 212 | 66.78 | 102.24 | 0.000 | 0.091 |

Записей с высокой долей p2p-артефактных эпох после финального QC обнаружено не было.

## Ограничения

Данный пайплайн решает задачу технической подготовки данных и базового контроля качества. Он не устраняет все возможные методические ограничения исследования.

Ключевые ограничения:

- данные ЧМТ и контрольной группы получены из разных источников;
- для клинических записей ЧМТ состояние глаз может быть указано менее детально, чем в контрольном датасете;
- различия в длительности записей могут влиять на стабильность последующих признаков;
- амплитудный QC используется как дополнительный аудит качества после ICA/ICLabel и AutoReject, а не как основной автоматический фильтр;
- унификация каналов повышает сопоставимость групп, но может приводить к исключению части доступной пространственной информации;
- дальнейшая интерпретация результатов должна учитывать возможные различия в протоколах регистрации и источниках данных.
